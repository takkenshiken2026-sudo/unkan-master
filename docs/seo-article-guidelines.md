# SEO記事テンプレート運用ルール

資格試験サイト向けSEO記事を、AI丸投げの薄い記事にしないための共通ルールです。新規記事だけでなく、既存記事のリライト時もこの形式に合わせます。

## 基本方針

- **知識（/terms/）と行動（/articles/）を混ぜない。** 立ち位置は [content-positioning.md](./content-positioning.md) を正本とする。
- 公式情報、一次情報、更新日を必ず確認する。
- 断定できない内容は断定しない。年度、受験資格、手数料、合格基準、免除制度、法改正は特に注意する。
- 記事ごとに検索意図を1つ決める。複数の検索意図を詰め込みすぎない。
- 汎用説明だけで終わらせず、そのサイト内の過去問、用語解説、復習機能、関連記事へつなげる。
- 「誰が確認したか」「何を根拠にしたか」「いつ確認したか」をページ上に出す。
- 毎回の更新では、本文を無理に増やすより、事実確認、内部リンク、FAQ、最新年度情報の見直しを優先する。

## 試験ガイドの本数（テンプレート標準）

本番サイトでは、試験ガイド（`guide_articles.csv`）を **100本以上** 用意することをテンプレート標準とします。用語300件以上と同様、複数資格・複数検索意図をカバーするための最低ラインです。

- サンプルテンプレート同梱分（5本）は動作確認用です。本番公開前に不足分を追加してください。
- 1記事 = 1検索意図を基本にし、同じ意図を複数記事に分割しすぎない。
- 記事追加後は `python3 tools/build_all.py` を実行し、`articles/` と `articles/index.html` を再生成する。
- 100本未満のまま公開する場合は、意図的な段階公開であることを `revision_note` または運用メモに残す。

## アフィリエイト記事（試験ガイド内・10本目安）

本番サイトでは、**ASP / 商品 URL が用意できたアフィリエイト記事を10本前後** 公開することを目安とします。**リンク未設定の slug は CSV 行も HTML も作らない**（`scaffold_affiliate_article.py --append` も ASP URL 必須）。別URLにせず **`articles/{slug}/` として試験ガイド一覧・関連記事から辿れる** 形を正とします。

### 作成条件（必須）

- **Amazon / A8 / afb 等の ASP URL が確定していること**（`related_links` の `https://...:ラベル`、または brief の `products.*_url`、本文 CTA）
- URL 未用意のテーマは **ブリーフ YAML のみ** 残してよいが、`guide_articles.csv` への追記と公開 HTML 生成は行わない
- 行だけ先に存在する場合は **`content_status=draft`** とし、ASP URL 確定まで非公開のままにする
- 例外: `affiliate-free-vs-paid-study` のように **asp=internal**（収益リンクなし・内部導線のみ）と明示したテーマのみ

### 識別ルール（CSV）

- `tags` に **`アフィリエイト`** を含める（**リンク済み行のみ**カウント。`validate_csv.py` が ASP リンク有無を検査）
- `genre` は原則 **`独学対策`**。模試・直前向けは `過去問活用` / `直前・当日`、学習設計からの導線用は `学習計画` も可（12ジャンル以外は不可）。
- 1記事 = 1検索意図（参考書総合・通信講座比較・塾 など）。10本を超える細分化は、重複キーワードより **更新・ASP案件** で差別化する。

### コンテンツ・法務（公開ページ）

- **【PR・広告】などの PR 定型表記は記事本文・表・一覧に載せない**（読者向けコンテンツのみ）。ASP・報酬条件は `original_note`（非公開）に記録する。
- **価格・料金は ASP / 公式 URL を開いて最新を確認**してから brief・本文に反映する（テンプレ数値・他記事からの転記禁止）。詳細は [affiliate-article-rules.md §3](./affiliate/affiliate-article-rules.md#執筆の必須原則必ず運用)。
- **本文は記事ごとにオリジナル執筆**する。テンプレの機械置換・他 slug 流用・一括生成した量産文は公開しない（同上 §3）。
- 価格・キャンペーンは **`fact_checked_at` と本文の確認日** を揃え、断定しすぎない。
- 外部リンク（A8.net・afb・Amazonアソシエイト等）は `related_links` の `https://...:ラベル` または将来の CTA 列で管理。公式サイトURLと混同しない。
- `primary_sources` は **公式・一次情報** 向け。ASPリンクは主な参照元に入れない。
- `original_note`（非公開）に ASP 名・プログラムID・報酬条件のメモを残す。

### 内部リンク（マネタイズ導線）

- **送り元:** 学習計画・独学対策・過去問活用など **15本前後** から、該当アフィリ記事へ `related_links` で双方向リンク。
- **送り先:** 各アフィリ記事から過去問・用語・学習計画へ戻す（無料演習との組み合わせを書く）。
- 全ページフッターへの一括貼り付けはしない。学習段階に合った **1〜2本** に絞る。

### ASP（テンプレ想定）

- **A8.net / afb** … 通信講座・予備校・高単価サービス（収益の柱）
- **Amazonアソシエイト** … 参考書・問題集（具体ISBN・信頼補助）

### 本数・検証

- 本番目安: **ASPリンク済み 10本**（立ち上げは1本ずつ追加。不足時は `validate_csv.py` が警告）
- slug 例: **[guide-article-catalog.md](./guide-article-catalog.md)** の「アフィリエイト記事（10本目安）」
- 10本未満でも、**リンク未設定の placeholder 行を増やさない**

## 推奨記事ジャンル

試験ガイドの `genre` は **`site-config.json` の `guideArticleGenres` で定義した12ラベルのみ**（MECE・資格共通）。フェーズ表・100本の増やし方は **[guide-article-genres.md](./guide-article-genres.md)**、slug 一覧は **[guide-article-catalog.md](./guide-article-catalog.md)** を正本とします。

**制度:** 試験概要 / 受験・申込 / 合格・難易度 / 出題・形式  
**学習:** 学習計画 / 独学対策  
**演習:** 過去問活用 / 分野別対策 / 用語ハブ活用法 / 復習・苦手克服  
**直前:** 直前・当日  
**横断:** 注意点・更新

本番では **100本以上** を推奨します（`validate_csv.py` が未満で警告）。記事を増やすときはジャンルを増やすより、同ジャンル内で検索意図を分割し、**分野別対策** を `fields` ごとに複製するのが効率的です。

## 参考記事の再現ルール

用語詳細記事を作るときは、次の記事型を基準にします。

- **執筆テンプレ・必須列:** [glossary-term-template.md](./glossary-term-template.md)（**全用語を詳細記事として公開**。`tools/glossary_term_rules.py` + `validate_csv.py` が ERROR で検証）
- **雛形生成:** `python3 tools/scaffold_glossary_term.py --term 用語名 --category 分野名`
- 参考URL: `https://YOUR-DOMAIN.example/terms/g-example.html`（用語ページの URL 例）
- 参考にするもの: 構成、見出し順、情報量、表で比較する説明、FAQ、関連ページへの導線
- そのまま写さないもの: 文章、試験名、固有の制度説明、公式URL、数値、日付
- コンテンツ本文幅: PC表示では約860pxを上限の目安にする
- **用語・問題への HTML 図解:** 色・レイアウト・コンポーネントは [term-diagrams.md](./term-diagrams.md) **§6 デザイン仕様** に従う（`site-pages.css` の `.term-diagram-*`、CSS 変数のみ）

再現時は、次の順番を基本にします。

1. リード文
2. 目次
3. この記事の信頼性について
4. この記事でできること
5. 本文見出し 4〜7個
6. よくある質問
7. 記事の基本情報
8. 公式情報の確認
9. 関連記事

本文は、単なる説明文だけでなく、比較表、頻出の誤り、選択肢で迷いやすい点を入れます。特に区分、義務、期限、数値、対象範囲の違いは、文章よりも表で整理することを優先します。

AIに依頼するときは、元記事URLだけでなく、対象資格名、対象分野、検索意図、必ず確認する公式情報、使ってよい内部リンクを一緒に渡します。元記事の見出しと説明順は参考にしてよいですが、本文は対象資格に合わせて独自に書き直します。

「この記事の信頼性について」は末尾に回さず、目次の直後に配置します。ただし冒頭で読者の離脱を招かないよう、公開ページでは執筆、確認、事実確認日、主な参照元を短くまとめ、詳しい運営説明は別ページへ分けます。更新方針は公開表示せず、内部管理項目として扱います。

「この記事の信頼性について」と「記事の基本情報」は、文章だけで並べず表形式にします。信頼性表は冒頭寄り、基本情報表は本文とFAQの後、公式情報の確認の前に置きます。

## フォント・表デザインルール

- フォントは `Noto Sans JP` を基本にする。
- 本文サイズは16px、行間は1.75〜1.9を基準にする。
- 本文、リード文、表の本文セルは薄く見えないよう、`font-weight: 400` 以上を基本にする。リード文や要約ボックスは `font-weight: 500` まで上げてよい。
- 記事タイトルは24px前後、本文見出しは19px前後を基準にする。
- 文字サイズは増やしすぎず、原則として次のスケールから選ぶ。
  - 11px: 小さなラベル、カテゴリ、補助バッジ
  - 12px: キャプション、更新日、メタ情報
  - 13px: ナビ、補足リンク、細かいUI
  - 14px: サブテキスト、カード説明、表内の補助文
  - 16px: 本文、リード文、表の本文セル
  - 19px: 記事本文のh2
  - 24px: 記事タイトル
- 新しい文字サイズを追加する場合は、既存スケールで代替できない理由があるときだけにする。
- 比較表・整理表は、見出し行をダークグレー `#2b2b2b`、文字を白にして、本文中で視認性を高くする。
- 信頼性表・記事基本情報表などのメタ情報表も、見出しセルはダークグレー `#2b2b2b`、文字は `#ffffff` にして、項目名をはっきり見せる。
- 表デザインは記事・用語ページだけでなく、フッターから遷移できる静的ページ（このサイトについて、プライバシー、関連リンク等）でも共通にする。
- 表で色分けをする場合は、添付色の `#333333` を基準色として使う。重要な見出し、区分名、比較表の軸、強調したいセルに使い、文字色は `#ffffff` にする。
- 色分けは多色にしすぎず、`#2b2b2b`、`#f8fafc`、白背景を基本にする。追加色は正誤、注意、合格・不合格など意味が明確な場合だけ使う。
- 表の線は薄いグレーの境界線にし、強い色やカラフルな装飾は使わない。
- 表はスマホでも横幅に収まるよう、セル内の文章を短くし、必要なら項目を分ける。

## 文字色ルール

- 基本の文字色は増やしすぎず、原則として次の5系統から選ぶ。
  - `--text` / `#121212`: 見出し、重要語、主本文
  - `--ink` / `#333333`: 通常本文、リード文、リンク
  - `--text2` / `#555555`: 補足説明、注釈、コピーライト
  - `--text3` / `#999999`: 更新日、弱いメタ情報、非主要ラベル
  - `--table-head-bg` / `#2b2b2b`: 表見出しセルの背景
- 比較表・整理表の見出し行は、背景 `#2b2b2b`、文字 `#ffffff` にする。
- 信頼性表・記事基本情報表の見出しセルは、背景 `#2b2b2b`、文字 `#ffffff` にする。
- 表の本文セルは `--text` / `#121212` を使う。
- 静的ページ内の表も、見出しセルは背景 `#2b2b2b`、文字 `#ffffff`、本文セルは `#121212` を使う。
- リンク文字は本文色に近い `#333333` を基本にし、下線でリンクだと分かるようにする。
- フッターリンクと静的ページ（このサイトについて、プライバシー、関連リンクなど）の本文も `#333333` を基本にし、薄いグレーにしすぎない。
- フッターのコピーライトや補足文は `#555555` までに留め、主要リンクは `font-weight: 700` を使う。
- 赤、青、緑、オレンジなどの意味色は、正誤表示、注意喚起、カテゴリバッジなど意味が明確な場合だけ使う。

## アクセント色・試験ガイド一覧UI

ブランド色は `site-config.json` の `theme.accent` / `theme.accentText` を正とし、`tools/site_config.py` が `site-theme.css` で次のトークンを生成します。

- `--accent` … ブランド色本体（ロゴマーク等）
- `--accent-soft` / `--accent-border` / `--accent-emphasis` … ラベル・強調・ホバー用（`color-mix` で自動生成）
- `--accent-hover-surface` / `--accent-hover-border` / `--accent-shadow` … カード・関連リンクのホバー

### 色の付け方（テンプレート標準）

| 要素 | ルール |
|------|--------|
| 本文・リード | `#333333`（`--ink`）。accent で全文を染めない |
| 文中の `<strong>` | `--accent-emphasis`（accent と ink の中間トーン） |
| カテゴリ・見出し番号・分野バッジ | 薄い背景 + 細い枠 + `--accent-emphasis` 文字（ベタ塗りの濃色ピルにしない） |
| 試験ガイド一覧の**カード本体** | 白背景・共通グレー枠。ジャンル別の背景色・上線は**付けない** |
| 試験ガイド一覧の**ジャンルラベル**（バッジ・フィルタチップ） | `guideArticleGenres` の `style` 別（制度=青 / 学習=緑 / 演習=オレンジ / 直前=紫）。`site-pages.css` の `data-genre-style` |
| カード・関連リンクのホバー | ごく薄い accent トーン。派手な影やジャンル色の全面塗りはしない |
| ロゴマーク（`.site-page-mark`） | `--accent` 塗りつぶし可（識別用） |

色味は「弱すぎず強すぎず」の中間を標準とします。変更時は `site-pages.css` の mix 比率と `site-theme.css` をセットで見直し、記事・用語・試験ガイド一覧で統一します。

### 試験ガイド一覧（`articles/index.html`）

- 読者向け: 記事検索、ジャンル絞り込み、記事カード、公式情報への導線のみ。
- **載せない**: 「共通テンプレの増やし方」「記事を増やすときの例」など、運用者・編集者向けの説明セクション。
- meta description に「テンプレート」などの運用語を入れない（読者向けの説明にする）。

## 公開ページの境界（読者向け / 運用者向け）

受験者・学習者にとって価値のない情報は、**表にも本文にも一覧にも出さない**。

### 公開ページの表に載せてよいもの

**信頼性表**

- 執筆、確認、事実確認日、主な参照元

**記事の基本情報（試験ガイド）**

- ジャンル、タグ（読者の整理用。`アフィリエイト` は CSV 内部識別用のため **表・一覧には出さない**）

**記事の基本情報（用語）**

- 対象試験、分野、重要度、関連タグ、法令・根拠（値がある場合のみ）

### 公開ページに絶対出さないもの（CSV内部用）

| 項目 | 扱い |
|------|------|
| `update_policy`（更新方針） | CSV・監査のみ。`validate_generated_seo.py` が検出するとエラー |
| `original_note`（独自メモ） | CSV・監査のみ |
| 用語ページの「検索意図」「記事種別」行 | 生成しない |
| 試験ガイド一覧のテンプレ運用説明 | 生成しない |
| 次回確認日の具体的日付を読者向けに明記 | `next_review_at` は内部列 |

### 本文について

- `section_*` や FAQ に、CSV追加手順・テンプレ拡張・SEO運用の話を書かない（サンプルに残っている場合は本番前に差し替え）。
- 運用メモは `revision_note` / `original_note` に書き、公開HTMLに流さない。

## 必須構成

各SEO記事は、原則として次の順番にします。

1. タイトル
2. リード文
3. 目次
4. この記事の信頼性について
5. この記事でできること
6. 本文見出し 5〜7個
7. よくある質問
8. 記事の基本情報
9. 公式情報の確認
10. 関連記事

## タイトルルール

- 検索キーワードを自然に入れる。
- 「◯◯試験 + 知りたいこと」の形を基本にする。
- 誇張しない。「完全攻略」「絶対合格」など根拠のない表現は避ける。
- 30〜45文字程度を目安にする。

例:

- `◯◯試験の概要と最初に確認するポイント`
- `◯◯試験を独学で進める学習ロードマップ`
- `◯◯試験の過去問の使い方と復習の進め方`

## リード文ルール

- 誰向けの記事かを明確にする。
- 読むと何が分かるかを書く。
- 不確かな制度情報を断定しない。
- 80〜140文字程度を目安にする。

例:

```text
独学で合格を目指す場合は、教材を増やす前に出題範囲と復習の仕組みを決めておくことが大切です。
```

## 目次ルール

- 目次は必ず表示する。
- 「この記事の信頼性について」「この記事でできること」「よくある質問」も目次に含める。
- 本文見出しは5〜7個を基本にする。
- 見出しだけで記事の流れが分かるようにする。
- 目次項目が多い記事では、PC表示は2列、スマホ表示は1列にする。

## 本文見出しルール

本文見出しは、次のように読者の行動順に並べます。

1. 最初に確認すること
2. 全体像・前提
3. 具体的な進め方
4. 復習・確認方法
5. 注意点
6. つまずきやすいポイント
7. 次にやること

見出し例:

- `独学前に公式情報を確認する`
- `全体像を分野に分ける`
- `過去問で現在地を確認する`
- `復習を学習計画に入れる`
- `直前期は増やさず絞る`
- `独学でつまずきやすいポイント`
- `学習記録を残して再訪問につなげる`

本文の主要見出しには、参考サイトのように小さな番号バッジを付けて、目次と本文の対応を分かりやすくします。番号を付けるのは本文見出しだけにし、「この記事の信頼性について」「この記事でできること」「よくある質問」「記事の基本情報」「公式情報の確認」などの共通ブロックには付けません。

用語詳細記事では、「定義と基本理解」のような定義ブロックを薄いグレー背景の囲みで少し強調します。本文全体は装飾しすぎず、見出し番号、比較表、定義ブロックで読みどころが分かる程度に留めます。

## 本文分量ルール

- 各見出し本文は180〜300文字程度を目安にする。
- 1記事あたり本文合計は900文字以上を目安にする。
- 重要記事は1,500〜2,500文字程度まで増やしてよい。
- 同じ文型を繰り返さない。
- 箇条書きだけで終わらせず、理由や使い方を書く。

悪い例:

```text
過去問を解きます。間違えた問題を復習します。用語を確認します。
```

良い例:

```text
基本事項を少し確認したら、早めに過去問を1年度分解いてみます。点数だけで判断せず、読めなかった用語、迷った選択肢、時間がかかった分野を記録してください。ここで見つけた弱点が、次に読む用語解説や復習対象になります。
```

## 信頼性ブロック

公開ページの信頼性表には、次の行を入れます。

- 執筆: 執筆者名と、必要に応じて執筆者プロフィール
- 確認: 確認者名と、必要に応じて確認者プロフィール
- 事実確認日: 公式情報やCSV内容を確認した日
- 主な参照元: 公式サイト、試験要項、法令、一次情報など

`original_note`（独自メモ）、`update_policy`（更新方針）、用語ページの「検索意図」「記事種別」は CSV 上の内部管理用です。**公開ページの表には表示しません。** 試験ガイド一覧に「テンプレの増やし方」など運用説明のセクションも置きません。

`update_policy` は公開ページには表示しません。CSV上の内部管理項目として残し、「どの条件で見直すか」を管理します。具体的な次回確認日は `next_review_at` で内部管理します。

良い例:

```text
試験要項や公式ページが更新されたタイミングで本文と参照元を見直します。
```

避ける例:

```text
次回は2026年6月17日に確認します。
```

記事ページの `primary_sources` は `ラベル|URL` 形式で入力します。用語詳細ページは、個別の参照元列がない場合でも `site-config.json` の `externalLinks`、または `officialOrganization` から生成される公式リンクを「主な参照元」として表示します。

```csv
試験実施団体（公式）|https://example.com/
```

本番サイトでは `example.com` を必ず実際の公式URLに差し替えます。

生成後は `python3 tools/build_all.py` を実行し、`tools/validate_generated_seo.py`・`tools/validate_internal_links.py`・`tools/validate_public_content.py` の結果を確認します。信頼性表の欠落、リンク切れ、公開ページへの運用向け表示がある場合は修正してから公開します。

## テンプレート標準のSEO実装

本テンプレートでは、次も標準で生成します。

- 用語ページ: `DefinedTerm` を中心とした JSON-LD（`WebPage` の `mainEntity`、`FAQPage`、`BreadcrumbList` を併記）
- 分野別用語ハブ: `terms/field-xxx/index.html`（`site-config.json` の `fields` に対応）
- `sitemap.xml`: 全公開URLに `lastmod`（試験ガイドは CSV の `fact_checked_at` 等、それ以外は生成ファイルの更新日）
- 関連リンクのフォールバック: `related_terms` や `related_links` が不足している場合、同分野の用語・同ジャンルの試験ガイドを**存在確認済み**の候補から自動補完
- 用語ページの目次: **本文が生成される見出しだけ**を目次に含める（空の「法令・根拠」などを目次に載せない）

## この記事でできること

`user_intent` には、読者が記事を読んで達成できることを書きます。

`action_items` には、読後に取る行動をセミコロン区切りで3〜6件入力します。

例:

```csv
公式情報を確認する;出題範囲を分野に分ける;過去問を1年度分解く;間違えた問題を復習リストへ残す;関連用語を読む
```

## FAQルール

- FAQは2件以上を基本にする。
- FAQは最初から開いた状態で表示する。
- 質問は検索ユーザーの不安に合わせる。
- 回答は100〜200文字程度を目安にする。
- 公式情報が必要な内容は、公式情報確認を促す。

例:

```text
Q. 独学では何から始めればよいですか？
A. まず公式情報で試験の範囲と日程を確認し、その後に過去問を使って現在地を把握します。最初から完璧に理解しようとせず、分からなかった用語や分野を復習対象として残すことが大切です。
```

## 関連記事ルール

- `related_links` は2件以上を推奨する。
- 記事の最後に次に読むページを置く。
- 関連記事は `slug:表示名` 形式で入力する。

例:

```csv
exam-overview:試験概要;study-plan:学習計画;past-question-strategy:過去問の使い方
```

## 内部リンク（リンク切れゼロ）

内部リンクは SEO と回遊の要です。**公開ページにリンク切れを出さない**ことをテンプレート標準とします。

### CSV で守ること

- `guide_articles.csv` の `related_links` に書く slug は、**同じ CSV に存在する slug のみ**（存在しない slug はビルド前に `validate_csv.py` がエラー）。
- `glossary_terms.csv` の `related_terms` は、**同じ CSV に存在する用語名を2件以上**（未登録語は **ERROR**、リンク化されない）。
- 外部URLは `http://` / `https://` のみ。内部向け slug を URL 列に混ぜない。

### 生成時のフォールバック

- 試験ガイド: `related_links` が不足している場合、同ジャンル・同タグの記事や定番 slug を**存在確認済み**の候補から自動補完する。
- 用語: `related_terms` が不足している場合、同分野の用語を自動補完する。CSV で指定した用語は必ずリンク化され、未登録語は出力しない。

### ビルド時の必須チェック

`python3 tools/build_all.py` では次を必ず実行します。

1. `tools/validate_csv.py` … CSV 段階で内部リンク先の存在確認、公開禁止の見出し・本文の検出
2. `tools/validate_generated_seo.py` … 信頼性表・SEO 構成
3. `tools/validate_internal_links.py` … 生成後の全 HTML の `href` を走査し、相対パス・同一ページアンカーの解決を確認
4. `tools/validate_public_content.py` … 運用者向け文言・禁止表行の再検出

いずれかでエラーになったら**公開・デプロイしない**。修正後に再ビルドします。

`index.html` の `/#past` など SPA 用ハッシュは、ルーティングで処理する既知の名前のみ許可します。用語解説は `terms/index.html`（静的一覧）に集約し、`#glossary` は使いません。新しいハッシュを増やす場合は `tools/validate_internal_links.py` の許可リストも更新してください。

## 更新管理ルール

`guide_articles.csv` では次の列を使って更新頻度を管理します。

- `last_reviewed_at`: 最後に記事内容を確認した日
- `next_review_at`: 次に確認する予定日（内部監査用。公開ページや `update_policy` には書かない）
- `source_checked_at`: 公式情報を確認した日
- `content_status`: `draft`, `review_needed`, `published`, `archived`
- `revision_note`: 何を更新したか

月1回を目安に次を実行します。

```bash
python3 tools/audit_article_freshness.py
```

レビュー対象がある場合にCIで失敗させるには次を使います。

```bash
python3 tools/audit_article_freshness.py --fail-on-review
```

## CSV入力フォーマット

新規記事は **[guide-article-template.md](./guide-article-template.md)**（執筆テンプレ）と `python3 tools/scaffold_guide_article.py` を参照してください。

最低限、次の列を埋めます。

```csv
slug,genre,title,meta_description,lead,priority,tags,author_name,author_profile,reviewer_name,reviewer_profile,fact_checked_at,primary_sources,original_note,user_intent,action_items,update_policy,last_reviewed_at,next_review_at,source_checked_at,content_status,revision_note,section_1_heading,section_1_body,section_2_heading,section_2_body,section_3_heading,section_3_body,section_4_heading,section_4_body,section_5_heading,section_5_body,faq_1_question,faq_1_answer,faq_2_question,faq_2_answer,related_links
```

本文を厚くする記事では、追加で次を使います。

```csv
section_6_heading,section_6_body,section_7_heading,section_7_body
```

## 既存記事リライト手順

1. 記事の検索意図を1つに決める。
2. 公式URLと確認日を入れる。
3. 著者・確認者・事実確認日・参照元を入れる。
4. 目次に入る見出しを5〜7個へ整理する。
5. 各見出し本文を180〜300文字程度に増やす。
6. FAQを2件以上にする。
7. 関連記事を2件以上入れる。
8. `revision_note` に変更内容を書く。
9. 監査とビルドを実行する。

```bash
python3 tools/validate_csv.py
python3 tools/audit_article_freshness.py
python3 tools/build_all.py
```

## 公開前チェックリスト

- 公式URLが `example.com` のままではない。
- 試験名、年度、日程、手数料、受験資格、合格基準を確認した。
- `fact_checked_at` と `source_checked_at` を更新した。
- 目次が表示され、目次のアンカー先がすべて存在する（空セクションを目次に含めていない）。
- FAQが開いた状態で表示されている。
- 関連記事リンクが2件以上ある。
- 記事本文が短すぎない。運用者向けの見出し・段落が残っていない。
- 信頼性表に運用向け行（独自メモ・更新方針など）がない。
- 試験ガイド一覧にテンプレ運用のセクションがない。
- アフィリエイト記事は `tags` に `アフィリエイト` があり、**ASP リンク設定済みの行のみ** HTML 公開（本番10本目安）。
- `related_links` の slug がすべて CSV に存在する。
- `related_terms` は登録済み用語を2件以上にし、未登録語の ERROR を解消した。
- 全用語で `article_title`・`term_detail_body`（120字以上）・`exam_points`（2項目以上）・**FAQ 4組**・例題が埋まっている。
- 用語記事の文体はです・ます調・短文・専門語の補足（[glossary-term-template.md](./glossary-term-template.md) の文体ルール）。
- `revision_note` に更新内容が残っている。
- 次がすべて成功すること:

```bash
python3 tools/validate_csv.py
python3 tools/audit_article_freshness.py
python3 tools/build_all.py
```

`build_all.py` は内部で `validate_generated_seo.py`・`validate_internal_links.py`・`validate_public_content.py` も実行する。**検証エラーが1件でもあれば公開しない。**
