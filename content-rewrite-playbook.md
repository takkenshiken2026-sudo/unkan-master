# Webサイト記事 品質リライト 指示書（汎用テンプレート）

このまま AI エージェント（Claude Code 等）に渡せる、サイト横断で使える
記事リライトの作業指示です。`{ }` の箇所だけ自分のサイトに合わせて埋めて
ください。元サイト固有の事情に依存しない設計にしています。

---

## 0. 使い方
1. 「サイト設定」を埋める。
2. まず **フェーズ1（診断）** を実行させ、問題の棚卸しレポートを受け取る。
3. レポートを見て、どのフェーズまでやるかを決める。
4. 各フェーズは「対象を分割 → 並列エージェント → 機械検証 → ビルド → 反映」で回す。
5. **公開（デプロイ）は必ず人間が承認**してから行う。

## 1. サイト設定（埋める）
- サイトの題材 / 想定読者: `{例: ○○資格試験の受験者}`
- 記事データの所在: `{例: data/articles.csv / Markdown / CMS API}`
- 1記事のフィールド構成: `{title, meta_description, lead, section_N_heading/body, faq, related_links 等}`
- ビルド方法: `{例: python build.py / npm run build}`
- 検証コマンド: `{例: リンク切れ・スキーマ・文字数チェックのスクリプト}`
- 公開方法: `{例: main へ push で自動デプロイ}`
- 「変更してはいけない」フィールド: `{例: 出典URL(primary_sources), 商品ID, 著者名, related_links のslug}`
- サイト共通の不変の事実: `{例: 数値・料金・制度の定義など、全記事で統一すべき値}`

## 2. 絶対ルール（全フェーズ共通）
- **事実・数値は創作しない**。裏が取れない事実は書かない。
- 数値・固有名詞・制度内容を変える時は **必ず一次情報（公式・公的機関・法令）を検索して裏取り**し、出典URLを報告する。
- **出典URL・商品ID・内部リンクのslug・著者情報・日付**などメタ情報は原則保持。
- 表・リスト・コードブロックなどの**構造を壊さない**（区切り行・列数・パイプ数を維持）。
- 編集は指定フィールドのみ。**キーの追加・削除をしない**。
- 出力は機械検証できる形式（例: `{"slug": "...", "fields": {...}}` の純粋JSON）で、前後に説明文を付けない。
- **デプロイは人間の承認後**。破壊的操作（記事削除・統合）は提案 → 承認 → 実行の順。

## フェーズ1｜診断（まず棚卸し。編集しない）
目的: 何が問題かを実データから洗い出す。レポートのみ作成。

調べること:
1. **読みにくさ**: 一文が長い／思わせぶり・煽り・謎かけの見出し／作り込んだ不自然な例え・必然性のない日時設定／見出しと本文の不整合。
2. **中身の薄さ**: 「勉強法・概要などメタ情報」ばかりで、読者が実際に知りたい中身（定義・数値・具体例・手順）を教えていない記事。
3. **重複**: 同じ事実を切り口だけ変えて繰り返す記事群。各記事に独立した検索意図・独自価値があるか。結論の数値が記事間で矛盾していないか。
4. **技術的な漏れ**: 本文・見出し・FAQ・メタに紛れた ファイル名(`*.html`)・URL断片・生のslug・社内用語・英語ジャーゴン。
5. **表記ゆれ**: 用語・数値・単位・中黒/記号の不統一、文字化け、語の重複。
6. **制約**: 公開記事数の上限、文字数の下限/上限、リンク切れなどの既存検証ルール。

出力: 問題の分類・該当記事リスト・重複マップ・推奨対応の優先順位。

## フェーズ2｜読みやすさリライト（非破壊・追記/書換）
各記事に対して:
- 一文を短く（目安 **1文40〜60字、最長でも80字**）。役所言葉・冗長表現を平易化。
- 見出しは**説明的に**。「主題 — 補足」の形式。思わせぶり/煽り/謎かけは禁止。
- 作り込んだ不自然な例え・根拠のない具体時刻/日付 → 自然な説明・手順に。
- 見出しと本文の数値・内容を一致させる。文字化け・重複・表記ゆれを修正。
- **事実・数値・表・リンク・固有名詞は維持**。

## フェーズ3｜中身の価値を上げる（要・出典裏取り）
「メタ情報だけ」の記事を、読者が実際に学べる内容にする:
- その分野で**実際に問われる/知りたい中身**（定義・区分・数値・具体例・手順・落とし穴）を、検索で一次情報を裏取りして追加。
- ラベルの羅列で終わらせない。具体値・比較・例・誤解しやすい点を入れる。
- 1記事につき「実内容セクション3〜4本＋進め方1本」程度を目安に、中身を主役にする。
- 改正されうる数値は**最新の改正版を検索確認**し、本文に「○○による／公式が正本」と明示。
- 報告に**裏取りした主要事実と出典URL**を必ず列挙させる。

## フェーズ4｜重複の統合（破壊的＝要承認）
- フェーズ1の重複マップを基に「残す/統合する/役割を明確化する」を決める。
- 統合時は**統合元の独自要素を統合先へ移植**し、**記事間の数値矛盾を一本化**。
- **いきなり削除しない**。サイトのリダイレクト機構（無ければ作る）で旧URL→新URLを noindex リダイレクト。
- 旧記事への**内部リンク（related_links・本文参照）を統合先へ張替**してからアーカイブ/削除。
- 公開記事数の上限・リンク切れ検証を必ず再実行。

## フェーズ5｜不要な英語・技術トークンの除去
読者向けフィールドから:
- ファイル名（`page.html` 等）→ 役割を表す日本語（例「公式の○○ページ」）。**実URLは出典欄に保持**。
- URL断片（`example.co.jp/path/` 等）→「公式サイト」「○○ページ」。
- 露出した生のslug → 記事名の日本語表記（**Markdownリンク `](../slug/)` の中は壊さない**＝境界判定で除外）。
- 社内用語・ジャーゴン・英略語 → 一般的な日本語に統一（固有名詞や CBT/PDF/URL 等の一般略語は残す）。

## フェーズ6｜ビルド/レンダリング側のバグも疑う
- 表示の崩れ（例: 普通の文が箇条書きになる）が**データではなく変換ロジック由来**のことがある。生データを確認し、原因がビルド側なら**ビルドを直す**（1記事ずつ手で直さない）。
- 修正後は**全記事を再生成**して影響範囲を確認。

## 実行パターン（効率と安全）
- **分割して並列化**: 記事を6本ずつ程度のバッチに分け、バッチごとに1エージェントを並列起動。各エージェントに「共通仕様書」を読ませてから作業させる。
- **共通仕様書**を1ファイル用意（守る制約・出力形式・検証項目）。プロンプトを短く保てる。
- **機械検証を必須化**（各エージェントの出力＆全体反映後の両方）:
  - JSONパース可・全キー保持・指定フィールド以外不変
  - 文字数下限/上限・一文長・表の列数/区切り
  - 禁止文字（例 `·`(U+00B7)）・残してはいけない語（社内用語・旧表記）がゼロ
  - 内部リンク切れゼロ・スキーマ検証パス
- **適用→ビルド→全検証→（承認）→デプロイ** を毎回の単位にする。
- バージョン管理: 作業用ブランチで開発・コミット、**公開ブランチへの反映は承認後**。コミットは変更理由が分かる粒度で。
